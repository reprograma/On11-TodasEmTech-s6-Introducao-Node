Perguntas

1) Qual a relação entre os métodos HTTP e o CRUD?

HTTP, Hypertext Transfer Protocol é um protocolo que permite a obtenção de recursos, como documentos HTML. É a base de qualquer troca de dados na Web e um protocolo cliente-servidor, o que significa que as requisições são iniciadas pelo destinatário, geralmente um navegador da Web.

CRUD, Create (criar), Read (ler) Update (atualizar) e Delete (apagar) são quatro principais funções usadas para interagir com aplicações de banco de dados. Muitas linguagens de programação e protocolos possuem sua versão equivalente de CRUD.

REST, Representational State Transfer (Transferência de Estado Representacional) é uma abstratação da arquitetura de software, principalmente de APIs web, que se baseia em limitações de design que, quando seguidas, é possível produzir aplicações com propriedades específicas.

API, Application Programming Interface (Interface de programação de aplicações) são um conjunto de padrões que fazem parte de uma interface e que permitem a criação de aplicações de forma mais fácil para desenvolvedores.

REST APIs se comunicam com clientes utilizando protocolo HTTP, que possui seus próprios métodos de manipulação de dados, como: GET, POST, DELETE, PUT, PATCH e entre outros. Há uma certa semelhança entre os métodos HTTP e as funções CRUD, como:

CRUD HTTP
CREATE POST/PUT
READ GET
UPDATE PUT/POST/PATCH
DELETE DELETE

REST APIs não são limitadas a funções CRUD. Por exemplo, desde que se use os métodos de HTTP apropriados, uma REST API pode permitir que clientes reiniciem um servidor, sendo que isso não é permitido por nenhuma das quatro funções CRUD.

2) Comente, com exemplos, a diferença entre o PUT e o PATCH.

PUT substitui um dado por completo (mesmo que esse dado não exista), e também pode ser usado para criar novos dados.

PATCH é usado para atualizar parcialmente um dado.

Exemplos: 
PUT /usuarios/1
{
    "usuaria": "helenadearaujo",
    "email": "helenadearaujo@email.com"       // novo endereço de email
}
        
PATCH /usuarios/1
{
    "email": "helenadearaujo@email.com"       // novo endereço de email
}
4) Defina o conceito de idempotência e como uma API pode ser idempotente

Em matemática e ciência da computação, a idempotência é a propriedade que algumas operações têm de poderem ser aplicadas várias vezes sem que o valor do resultado se altere após a aplicação inicial. Um método HTTP idempotente é um método HTTP que pode ser chamado muitas vezes sem resultados diferentes. Não importa se o método é chamado apenas uma vez ou dez vezes. O resultado deve ser o mesmo.

Em API’s REST, podemos concluir que os seguintes os verbos:

  • GET, PUT, DELETE, HEAD e OPTIONS são idempotentes.
  • POST não é idempotente.

Há métodos seguros e não seguros, seguros são aqueles que não alteram o estado da aplicação e não seguros que podem alterar.

Método Idempotente Seguro
GET Sim Sim
POST Não Não
PUT Sim Não
DELETE Sim Não
HEAD Sim Sim
OPTIONS Sim Sim
5) Cite alguns diferentes padrões de projetos de software

Há 23 padrões de acordo com GoF (Gang of Four), que de acordo com suas finalidades possuem 3 categorias:

  • Padrões de criação: são aqueles que abstraem e ou adiam o processo criação dos objetos. Eles ajudam a tornar um sistema independentemente de como seus objetos são criados, compostos e representados. Um padrão de criação de classe usa a herança para variar a classe que é instanciada, enquanto um padrão de criação de objeto delegará a instanciação para outro objeto.
    Exemplos:
    • Abstract Factory
    • Object pool
    • Builder
    • Factory Method
    • Prototype
    • Singleton
  • Padrões estruturais: se preocupam com a forma como classes e objetos são compostos para formar estruturas maiores. Os de classes utilizam a herança para compor interfaces ou implementações, e os de objeto ao invés de compor interfaces ou implementações, eles descrevem maneiras de compor objetos para obter novas funcionalidades.
    Exemplos:
    • Private class data
    • Adapter
    • Bridge
    • Composite
    • Decorator
    • Façade (ou Facade)
    • Business Delegate
    • Flyweight
    • Proxy
  • Padrões de comportamento: se concentram nos algoritmos e atribuições de responsabilidades entre os objetos. Eles não descrevem apenas padrões de objetos ou de classes, mas também os padrões de comunicação entre os objetos.
    Exemplos:
    • Chain of Responsibility
    • Command
    • Interpreter
    • Iterator
    • Mediator
    • Memento
    • Observer
    • State
    • Strategy
    • Template Method
    • Visitor
